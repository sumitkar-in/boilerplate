# Skill: Tenant-Scoped Data Access

## What

Every tenant-table query must take a `TenantContext` — never query a tenant table without it.

```ts
// Good — every query is tenant-scoped
contactsService.findAll(tenantContext, filters);

// Avoid — never query a tenant table without context
contactsRepository.find();
```

`TenantContext` (`apps/api/src/core/tenants/tenant-context.ts`) is resolved once per request — TODO: by `tenant-resolver.middleware.ts` (subdomain → `x-tenant-id` header → JWT claim), not yet built — and stored in `AsyncLocalStorage`, so any service can read it without threading it through every function call manually.

## When to use

Any time a service method reads or writes a tenant-schema table (i.e. anything under a module's `entities/`). Core/global tables (e.g. `core.tenants`, `core.feature_flags`) are the exception — those aren't tenant-scoped by definition.

## Minimal example

```ts
// <feature>.controller.ts — generated by generate-module.js, already wired:
@Get()
findAll(@CurrentTenant() tenant: TenantContext) {
  return this.billingService.findAll(tenant);
}

// <feature>.service.ts
async findAll(tenant: TenantContext) {
  // every query here is scoped by tenant.schemaName
}
```

Outside an HTTP request (e.g. inside a cron sweep), there's no `@CurrentTenant()` to inject — `TenantSweepService.getActiveTenants()` gives you the tenant list instead, and you pass each tenant's id/schema explicitly (see skills/cron-jobs/SKILL.md).

## Common mistakes

- **Calling `getTenantContext()` outside a request.** It throws if no `TenantContext` is in `AsyncLocalStorage` for the current async call tree — this happens if you call it from a cron job or a script instead of an HTTP handler. Use `TenantSweepService` there instead.
- **Querying a tenant table by raw schema name string instead of `tenant.schemaName`.** Don't hardcode `tenant_acme` anywhere — always read it off the `TenantContext` you were given.
- **Skipping `@CurrentTenant()` because "this query doesn't seem tenant-specific."** If the table lives in a tenant schema, it's tenant-specific by construction — the schema-per-tenant design (see the architecture doc §6) means there's no such thing as a tenant-agnostic query against a tenant table.
- **Passing `tenantContext.tenantId` around instead of the whole `TenantContext`.** Methods need `schemaName` too (and sometimes `enabledFeatures`/`role`) — pass the whole object, not just the id, so you don't have to thread extra parameters in later.
